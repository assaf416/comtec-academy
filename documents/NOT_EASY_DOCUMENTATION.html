🧩 "Not-easy" folder — M · UNFILE2PCC · UNSNDWEB

IBM i (AS/400) menu subsystem & AS/400↔WEB portability engine  •  folder example/not easy

The three heavy programs: an RPG menu driver, a CL DDS-build utility, and a 170-section COBOL WEB screen engine. Sources carry ASCII code with Windows-1255 Hebrew comments (decoded here to UTF-8).

🗂️ Files in this folder

Three unrelated-by-runtime but conceptually-linked programs from the AS/400↔WEB portability subsystem. Every program supports a dual code path: lines flagged X&FLTA compile for AS/400 (COBOL), lines flagged X&FLTU compile for UNIX.

FileTypeLanguagePurpose
M.rpg Interactive program (WORKSTN) RPG III / RPG 400 The main MENU driver for the "ME" application. Displays a 5250 menu, reads the user's choice and F-keys, checks authorization, logs activity, and launches the selected program or sub-menu. 68 subroutines.
UNFILE2PCC.cl Utility program CL (Control Language) DDS / file build utility. Reverse-engineers a physical file (DSPFD/DSPFFD), creates DDS source members A/F/HDR/X, builds PF objects, copies the data, and compiles+runs a generated header program. Build-time helper for the conversion pipeline.
UNSNDWEB.cbl Called subprogram COBOL (ILE / OPM) The WEB screen engine (Hebrew banner ניהול בניית מסכי WEB = "management of building WEB screens"). One 64,412-byte parameter, a big action-code dispatcher, XML build/parse, and data-queue transport to the web tier. 170 sections, ~9,200 lines.
⚠️ Ship-together warning (from the UNSNDWEB source header, "field-enlargement to K64" project): UNSNDWEB must be released together with UNPARTQU, UNQUEST, UNSCRWEB, UNSNDA2C, UNSNDC2X and the comms programs WBAPP, WBSRVR03, WBSRDQ. Because the shared buffer layout changed, deploying UNSNDWEB alone will corrupt the parameter exchange.

🔗 Relationships — who runs who

These three files are not a single call-chain. They belong to different tiers of the same system: an interactive menu, a build-time utility, and a server-tier engine.

📗 Menu tier — M.rpg

M reads menu/authorization/user files and, on a valid + authorized selection, CALLs the chosen application program (or opens a sub-menu). It does not call UNSNDWEB or UNFILE2PCC directly; the programs it launches are the general application catalogue.

📘 WEB tier — UNSNDWEB.cbl

UNSNDWEB is called by the web server tier (WBSRVR03/WBAPP). It exchanges XML with the browser front-end over a data queue (through WBSRDQ) and itself CALLs ~40 UN* helper programs to read screen/command definitions and build/parse the XML.

📙 Build tier — UNFILE2PCC.cl

UNFILE2PCC is a build-time utility: it calls UNCRTHDRK to generate DDS + a header-update program, then compiles and runs the generated UNUPDHDRK. It is invoked by the conversion tooling, not by M or UNSNDWEB at runtime.

📗 M.rpg — Menu driver

Program name & meaning

Program M — the interactive menu of the "ME" (menu) application. It shows a menu on a 5250 workstation, validates the option and the user's harshaot (authorizations), writes a usage log, and transfers control to the chosen program or sub-menu. RPG III / RPG 400, 68 subroutines.

📙 UNFILE2PCC.cl — DDS / file build utility

Program name & meaning

Program UNFILE2PCC — "UN[iversal] FILE to PC[C]". Given a physical file, it reverse-engineers the file's DDS and builds all the artifacts the conversion pipeline needs: a temporary source PF with four members (A / F / HDR / X), four PF objects, a copy of the data, and a compiled+run generated header program.

Uses MONMSG CPF0000 liberally to swallow "already exists / not found" conditions, so re-running the utility is idempotent. The only errors it surfaces are the two that block progress: source-PF creation failure and a missing input file.

📘 UNSNDWEB.cbl — WEB screen engine

Program name & meaning

Program UNSNDWEB = "UN[iversal] SEND to WEB". A single COBOL program with one entry parameter (a 64,412-byte buffer SND-DATA redefined by five parameter copybooks) and a big EVALUATE on an action code WB-KOD-PEULA. It builds XML for screens/fields/commands/grids/questionnaires/letters, exchanges it with the browser tier over data queues, and parses the XML reply back into the application's screen buffer.

DATA DIVISION — entry parameter & overlays

The program receives exactly one parameter, SND-DATA, and re-interprets it with five REDEFINES overlays — each backed by a copybook IN QWRKSRC. The detailed field layouts therefore live in those copybooks; what the source itself shows is the buffer and the redefines wiring.

CBL 000-MAIN — Action dispatcher
Called by: program entry (WBSRVR03 / WBAPP)
Calls: Z00-INITZ01-INIT-CLALIR21-SCREEN-CHECKC00/B00/E00/D00/T00/V00/X00…

Purpose: On first entry run Z00-INIT; coerce stray non-screen commands to CLOSE-FROM-WEB; then EVALUATE the action code and delegate. Loop-style actions (READ/SHELON/TEXTCTL/LETTER) are performed UNTIL SW-END-READ-ON.

IF SW-FIRST-TIME-ON  PERFORM Z00-INIT.
EVALUATE TRUE
  WHEN WB-KOD-PEULA-WRITE       PERFORM C00-TIPUL-WRITE
  WHEN WB-KOD-PEULA-READ        PERFORM B00-TIPUL-READ UNTIL SW-END-READ-ON
  WHEN WB-KOD-PEULA-SND-SHELON  PERFORM E00-SND-SHELON UNTIL SW-END-READ-ON

  *> ...folder, unmainda, tbl-tokef, memory-flex, close, other...
