קובץסוגשפהייעוד
saatcgsl.cbl
PROGRAM-ID SAATCGSL		 תוכנית נקראת (שירות) ILE COBOL עוטף / מפקח לפני ה-API של CreditGuard. מאמת ומקצץ את הפרמטרים של הקורא, מביא מספר רץ, מתרגם קודים פנימיים, קורא למנוע ה-RPGLE SAATCG, ופורק את התשובה בחזרה אל הקורא. שלושה פרמטרים: רשומת התנועה בגודל 703 בתים (קלט/פלט), בלוק אפשרויות בגודל 400 בתים (קלט), ובלוק תוצאה בגודל 200 בתים (פלט).
saatcg.rpgle
PROGRAM SAATCG		 תוכנית נקראת (מנוע) ILE RPGLE (פורמט חופשי) מנוע ה-API. בונה בקשת XML מסוג SOAP doDeal מתוך רשומת התנועה, שולח אותה ל-CreditGuard מעל HTTP (HTTPAPI), מנתח את תשובת ה-XML, ממפה את התוצאות בחזרה אל הרשומה, ומתעד גם את הבקשה וגם את התשובה. פרמטרי כניסה: SAATCGI (קלט), SAATCGO (פלט), ecptrnsv (רשומה בגודל 703 בתים).

קשרים — מי מריץ את מי

אפליקציה עסקית קוראת ל-SAATCGSL ומעבירה את רשומת התנועה בתוספת בלוק אפשרויות קטן (שפה 1=עברית/2=אנגלית וקונפיגורציה 1=גבייה/2=תביעה). SAATCGSL מביא מספר רץ באמצעות SAGETBKR, מקצץ כל שדה קלט באמצעות DCSTRING, אורז את הערכים לבלוקים עם קידומת אורך, וקורא למנוע ה-RPGLE SAATCG. SAATCG קורא את פרטי ההזדהות/כתובת ה-URL שלו מ-SAMAINPF, בונה את בקשת ה-SOAP doDeal מאותה רשומה בגודל 703 בתים, שולח אותה לשרת CreditGuard, מנתח את תשובת ה-XML, כותב את התוצאה בחזרה אל הרשומה ואל בלוק הפלט שלו, ומתעד את כל חילופי הדברים. השליטה חוזרת במעלה השרשרת ו-SAATCGSL פורק את התשובה עבור הקורא. הרשומה בגודל 703 בתים היא עמוד השדרה: COBOL מכנה את שדותיה DCCGE-xxx, ו-RPGLE מכנה את אותם בתים PTxxx.

תוכניות נוספות הנקראות לאורך הדרך: SAGETBKR (מספר רץ), DCSTRING (קיצוץ/דחיסת מחרוזת), SAATCGSL#P (שינוי session/SSID), TBERR (טקסט הודעת שגיאה), ו—בתוך SAATCG—CONVERT (המרת קידוד EBCDIC-424 ↔ UTF-8-1208) ותוכנית השירות HTTPAPI.

שם התוכנית ומשמעותו

SAATCGSL = SA (קידומת אפליקציית סדרת SA) + ATCG ("אשראית / CreditGuard") + SL (סיומת "סליקה" / שירות עוטף). זהו עוטף דק והגנתי לפני מנוע ה-API בשפת RPGLE SAATCG. תפקידו העסקי הוא לאפשר לכל תוכנית פנימית של IBM i להריץ תנועת כרטיס אשראי מבלי לדעת דבר על SOAP, HTTP, XML, פרמטרים באורך משתנה של RPGLE או המרת קידוד. הערת הכותרת (בעברית) מציינת את חמש האחריות שלו:

  • לפקח / לעטוף את הקריאה ל-API של CreditGuard.
  • לאמת את תקינות הפרמטרים.
  • לבנות את הפרמטרים עבור ה-API.
  • לקרוא ל-API.
  • לעבד את הפרמטרים החוזרים מה-API (בתוספת הוספת USER DATA, למשל ענף פוליסה).
